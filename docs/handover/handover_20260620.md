### Audio device hot-swap bug — FIXED & VERIFIED on MBA (build 20260620.2317)
UAT passed: built-in ⇄ AirPods/headphones switching works, and when the device disconnects the app auto-falls-back to the built-in mic mid-session, no restart. Also shipped + verified: a small caption under the recording capsule showing the live system-default input device (commit af1df67). Delivered to MBA via the `student-v1.1` pre-release DMG.

(historical detail follows)

On MBA, connecting/disconnecting AirPods Pro broke push-to-talk; user had to manually pick the input device + quit/relaunch to recover. Root cause: AudioCapture used one long-lived shared `AVAudioEngine`, and `AudioDeviceManager.configureInputDevice` is a NO-OP (engine always uses the system default device; actual device switching is an unimplemented TODO because setting kAudioOutputUnitProperty_CurrentDevice corrupts AVAudioEngine). When the default device changed, the stale engine's inputNode stayed bound to the old device (unhandled `AVAudioEngineConfigurationChange`) → silent recording failure until restart. Fix (commit 04be401): create a fresh `AVAudioEngine` per recording in `startRecording()` → always binds to current system default. **Compiles; NOT yet verified on real AirPods hot-swap — Johnson to test on MBA (needs the new build delivered there, e.g. via a GitHub release).** Note: engine follows SYSTEM default, so if macOS doesn't auto-switch default to AirPods the user still picks it in macOS Sound settings (the app's own device picker is cosmetic until the device-switching TODO is done).
